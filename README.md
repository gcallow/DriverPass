# DriverPass
CS 255 DriverPass Design Document Repository

The DriverPass project involved designing an online system for DriverPass, a company that wanted to help students better prepare for their DMV driving exams. The 
client wanted a web-based system that would allow customers to purchase training packages, access educational materials and practice tests, and schedule driving
lessons with instructors. The system also needed administrative features for managing accounts, appointments, and DMV information updates.

One thing I did particularly well was translating the business requirements into UML diagrams and system designs. I think my use case, activity, and class diagrams
clearly represented how different users would interact with the system and how the various components of the system would work together.

If I could revise one part of my work, I would spend more time refining my class diagram. As I continued working on the project, I realized there were additional 
relationships and classes, such as manager classes and payment processing components, that could have been organized more effectively. I would improve the diagram 
by simplifying some relationships and making the responsibilities of each class clearer.

I interpreted the user's needs by focusing on the requirements discussed in the business requirements document and thinking about how customers and employees would 
actually use the system. Features such as online scheduling, practice tests, account security, and DMV updates were all included because they directly addressed
the client's goals. Considering user needs is important because even a technically sound system can fail if it is difficult to use or does not solve the problems 
it was intended to address.

When designing software, I like to start by understanding the business requirements and identifying the people who will use the system and the tasks they need to 
accomplish. From there, I use diagrams such as use case, activity, and class diagrams to organize my ideas and visualize how the system should work. In the future,
I would continue using these modeling techniques along with gathering user feedback and creating prototypes early in the design process to ensure the final system 
meets both business and user needs.
